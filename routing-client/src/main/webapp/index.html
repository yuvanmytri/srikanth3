Istio Routing Mission - WildFly Swarm

This mission showcases the usage of Istio routing functionality, which can be used to create A/B tests, blue/green deployments for production releases, or simple load balancing.

Here we'll learn how to integrate routing into an Istio / WildFly Swarm application, using RouteRules and multiple deployments of a service.

View the defaults

Click "Invoke" in the UI; do this several times. You will notice that the services are currently load-balanced at exactly 50%. The responses from service A and B should be balanced by default at 50% by the Kubernetes cluster. This is not always desirable for different routing requirements. Sometimes it is important to slowly direct traffic to a new service over time. In this case, we can supply an Istio RouteRule to control load balancing behavior.

Apply a custom Istio RouteRule to control load balancing behavior.

Log in to your project's namespace, navigate to the mission source code on your local environment, and run the following command from your CLI:

oc create -f rules/load-balancing-rule.yaml

Note that the RouteRule defined in the file above uses labels "a" and "b" to identify each unique version of the service. If multiple services match any of these labels, traffic will be divided between them accordingly. Additional routes/weights can be supplied using additional labels/service versions as desired.

View the load-balanced behavior

Click "Invoke" in the UI; do this several times. You will notice that traffic is no longer routed at 50/50%, and more traffic is directed to service version B than service version A. Adjust the weights in the rule-file and re-run the command above (use new weights of your choice). You should see traffic adjust accordingly. Rules may take a few seconds to be updated in the Istio-system.

Conclusion

Congratulations! You now know how to direct traffic between different versions of a service using Istio RouteRules. This feature may be applied for any number of routing needs.
